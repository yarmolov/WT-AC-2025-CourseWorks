# Вариант 02 — Бронь аудиторий «Не занято?» — Роли и действия

Краткое описание

MVP: Room, Booking, User, Role. UI отображает календарь с доступными аудиториями и их расписанием; API: /rooms, /bookings, /schedule, /users. Приёмка (MVP): корректное создание бронирования с предотвращением конфликтов, возможность отмены и переноса.

Роли и основные действия

- Администратор
  - CRUD для пользователей
  - CRUD для аудиторий (Room)
  - Назначение ролей пользователям
  - Просмотр всех бронирований и расписания
  - Отмена любых бронирований
  - Просмотр статистики использования

- Преподаватель
  - Просмотр календаря аудиторий и расписания
  - Создание бронирования (Booking)
  - Просмотр своих бронирований
  - Отмена/перенос своих бронирований
  - Расширенные права на бронирование (больше часов)

- Студент
  - Просмотр календаря аудиторий и расписания
  - Создание бронирования с ограничениями
  - Просмотр своих бронирований
  - Отмена своих бронирований
  - Ограничения по времени бронирования (меньше часов)

Примеры сценариев

- Администратор создаёт Room, назначает роли → Преподаватель бронирует аудиторию на 2 часа → Студент видит занятое время в календаре.
- Студент пытается забронировать уже занятую аудиторию → система показывает конфликт и предлагает альтернативные варианты.
- Преподаватель переносит своё бронирование → система проверяет новое время на конфликты и обновляет расписание.
